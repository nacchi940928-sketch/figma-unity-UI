# 这个工程是做什么的

这是一套用于探索「Figma 设计稿」和「Unity 游戏 UI」之间能否互相转换、来回编辑的试验工程。

简单说：设计师在 Figma 里画界面，程序在 Unity 里用 Prefab 实现界面，两边改来改去之后，希望最终还能对得上。这个仓库就是在验证这件事能不能做、怎么做、哪些字段能同步、哪些做不到。

需要提前说明：**这里不是可以直接交给项目使用的正式工具**。它更像一份「可行性验证 + 原型代码 + 规范草稿」。如果要做成真正能上线的产品，还需要单独做程序开发，并把 Figma 插件正式交付出来。

---

## 我们想解决什么问题

游戏 UI 常见的工作方式是：UI 在 Figma 出稿，程序在 Unity 里搭 Prefab。实际开发中，两边都会改——策划改文案、程序调布局、设计师改视觉。如果没有统一的数据格式和同步机制，很容易出现「Figma 和 Unity 各改各的，最后对不上」。

这个工程尝试建立一条可往返的链路：

- **从 Figma 到 Unity**：把设计稿导出，在 Unity 里自动生成或更新 Prefab。
- **从 Unity 到 Figma**：程序在 Unity 里改完 Prefab 后，把改动导回去，让 Figma 上的设计稿跟着更新。

目前，「Figma 导入 Unity」这条主链路已经跑通并做过验收；「Unity 回写 Figma」也已经有原型，但还在持续打磨细节。

---

## 整体是怎么协作的

整个流程涉及四个角色，可以按顺序理解：

**第一步，设计师在 Figma 出稿。** 用 Figma 插件（我们叫它 Tool 1）选中一个界面根节点，导出一份完整的设计数据，通常是 XML 文件，旁边再配一批 PNG 图片。这份文件会记录界面上每一个图层的位置、大小、文字、颜色、布局等信息，每个节点都有一个不会变的 `id`，相当于身份证号。

**第二步，程序在 Unity 里导入。** 打开 Unity 工程，通过菜单 `Tools > Figma UI Importer`，选择刚才导出的 XML 和同目录的图片，Unity 会把它翻译成内部的中间结构（IR），再生成对应的 UI Prefab。生成后的每个节点上会挂一个 `IRBinding` 组件，记下它来自 Figma 的哪个 `id`，以后来回同步就靠这个对应关系。

**第三步，程序在 Unity 里继续加工。** 导入出来的 Prefab 往往只是视觉和布局的基础。程序通常会往上挂 Button、写交互脚本、改锚点做屏幕适配。这些 Unity 特有的东西不会写进 Figma 数据里，但导入工具提供了「合并（Merge）」模式：再次从 Figma 导入时，只更新设计相关的部分，Button 和脚本会尽量保留在原节点上。

**第四步，需要把 Unity 的改动同步回 Figma 时，用导出工具。** 通过 `Tools > Figma UI Exporter`，以当初那份 full 导出文件为模板，把 Prefab 里程序改过的位置、文案、可见性等字段合并进去，生成一份 `unity-export.xml`。然后设计师用 Figma 的另一个插件（Tool 2）上传这份文件，Figma 会按 `id` 找到对应图层并更新。

这样就形成了一个闭环：设计 → 实现 → 程序调整 → 回写设计 → 设计再改 → 再导入 Unity。

## 仓库里都有什么

这个仓库不只是一个 Unity 工程，而是把「规范文档、样例数据、原型代码、辅助脚本」放在一起，方便反复试验。

**`doc/` 目录**放的是各种规范和说明。包括 Figma 导出格式长什么样、Unity 怎么映射字段、坐标系怎么换算、哪些属性已经支持双向同步等。如果要做正式工具，这些文档就是需求基线。

**`UITest/` 是 Unity 验证工程**，版本是 2022.3 LTS。真正的核心代码在 `Packages/com.figma-unity.ui/` 这个 Package 里，版本还是 0.1.0 的原型阶段。导入和导出分别对应两个 Editor 窗口。生成好的 Prefab 放在 `Assets/UI/Generated/` 下面。

**`FIGMA mvp/` 和 `MergeMVP/` 是两套真实界面的试验数据。** 前者是 Roguelite 风格的主界面，后者是 Merge2 游戏棋盘界面，也是最近主要在迭代的样例。每个目录里通常有：Figma 全量导出的 XML、Unity 回导的 XML、配套 PNG 图片，以及多轮调试留下的 patches 记录。打开 Unity 工程可以看到对应的 `roguelite-main-screen.prefab` 和 `merge2-game-board-ui.prefab`。

**`figmajson/examples/`** 是更早阶段的验收样例，可以当作历史参考。

**`tools/`** 里有一些 Python 小脚本，用来离线对比 JSON/XML 差异、做格式转换、验证 Patch 合并逻辑，不参与游戏运行时。

**Figma 插件本身不在这个仓库里。** 这里只维护插件该怎么对接的规范，以及供插件和 Unity 共同消费的样例文件。

## 几个容易误解的地方

**Figma 导出的是「全量」，Unity 导出的是「增量补丁」。** Figma 插件会把设计稿上几乎所有字段都导出来；Unity 导入时则按配置选择性地用到其中一部分。Unity 往 Figma 回写时，并不是把 Prefab 整棵树重新描述一遍，而是在当初那份 full 文件上打补丁——没改到的 Figma 字段会原样留着。所以不要以为 Unity 导出的文件就是 Prefab 的完整镜像。

**Button、脚本这类 Unity 专属内容，不会进 Figma。** 它们靠 Merge 模式留在 Prefab 里。Figma 只关心视觉和布局层面的信息。

**复杂视觉效果在 Unity 里可能会被「烘焙」成图片。** 比如多层描边、渐变、阴影，Unity 侧不一定能一比一用组件还原，有时会生成 `_bakedPaint` 目录下的烘焙图来近似表现。这是探索阶段的折中方案。

**坐标系不一样，需要换算。** Figma 用左上角为原点，Unity UI 用 RectTransform 的锚点和轴心。文档 `doc/06-坐标系约定.txt` 里有详细约定。如果在 Unity 里自己新建的节点用了默认居中锚点，导回 Figma 时位置可能会偏，需要先把锚点设到左上角。

**图片靠文件名对应，不靠 Figma 内部 hash。** 导出数据里图片节点通常只记一个 `imageFile` 文件名，实际 PNG 放在同目录，由 Unity 按文件名去找。

## 目前已经能做什么

经过几轮样例界面的导入导出试验，下面这些能力基本可用：

- 从 Figma 导出 XML，在 Unity 里生成 Frame、文字、图片节点组成的 Prefab。
- 再次导入时用 Merge 模式更新布局与视觉，同时保留程序挂上去的组件。
- 从 Unity 导出位置、尺寸、旋转、缩放、可见性、文案、文字对齐等字段回 Figma。
- 按文件名同步图片资源。
- 部分自动布局（Auto Layout）可以导入；导出侧会把布局调整写回，但还不是完整的双向 Auto Layout 引擎。
- 字体通过 Font Mapping 配置映射到 TextMeshPro。

同时也明确了很多**故意不做或暂时做不到**的事：比如 Lua 绑定、UI 事件、增量 diff 更新、ScrollRect/Mask 完全一致、Figma 组件变体完整双向，以及面向生产的批处理和 CI 集成。这些留给正式工具阶段再考虑。

## 日常推荐怎么用

如果是第一次把一个界面从 Figma 引进 Unity，建议用「静态绝对坐标 + 全量替换（Replace）」的方式导入，先得到一份干净的 Prefab。程序在此基础上挂交互、调锚点。

需要把 Unity 的改动告诉设计师时，用默认的 Export 配置导出 `unity-export.xml`，再到 Figma 里用 Tool 2 导入。

之后设计师改了 Figma，程序想同步最新视觉时，改用「Visual Merge」模式再导入同一份 full XML，这样 Button 和脚本不会被冲掉。

## 和正式工具的关系

可以把本仓库理解成「先把路探清楚」：

- 格式能不能通、字段要不要补、Merge 策略是否合理、资源怎么约定——这些都在这里试过。
- `doc/` 里的规范文档，就是正式开发时的需求底稿。
- `com.figma-unity.ui` 是参考实现，API 和稳定性都还没到产品级。
- 样例 XML 和 PNG 是联调用的标准测试集。

正式工具需要在此基础上做工程化：版本管理、完整测试、错误处理、与项目流水线集成，以及 Figma 插件的正式发布。本仓库的价值在于降低正式开发的风险，而不是替代正式交付。
